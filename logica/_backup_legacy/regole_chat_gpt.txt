## 1️⃣ STRUTTURA GENERALE DEL LAVORO
- Ogni sistema (VMC, AC, riscaldamento, ventilazione, energia) è indipendente.
- Tre livelli collegati:
  1. **File .txt** → regole e spiegazione umana (“cosa deve fare”).
  2. **File .yaml** → logica operativa Home Assistant (“come lo fa”).
  3. **Plancia .yaml** → interfaccia Lovelace (“come lo si vede”).
- GPT deve garantire coerenza tra i tre livelli, evitando duplicazioni delle logiche core.

## 2️⃣ COME GPT DEVE RISPONDERE
1. **Usare il Canvas**: per YAML/Lovelace rispondere solo con il file completo nel canvas.
2. **Un file per volta**: il canvas contiene un solo file; altre note vanno fuori dal canvas.
3. **Nomi chiari e coerenti**: usare entità standard (sensor/switch/input_*) allineate ai file .txt.
4. **Logica invariata se non richiesto**: ottimizzare leggibilità/diagnostica ma non cambiare il comportamento senza richiesta.
5. **Spiegare le modifiche**: indicare sinteticamente cosa è stato fatto e perché.
6. **Controllo di coerenza**: per logiche → confrontare col file `.txt`; per interfaccia → seguire `core/regole_plancia.md`.

## 3️⃣ CONVENZIONI OPERATIVE
- Prefissi modulo: `1_vent`, `2_vmc`, `3_heating`, `4_ac`, `5_energy_pm`, `6_surplus`, `9_debug_test`.
- Manual override: `input_boolean.<modulo>_manual`, `input_select.<modulo>_manual_mode`, `timer.<modulo>_manual_timeout`.
- Hook: usare i nomi definiti in `core/regole_core_logiche.md` (es. `hook_vmc_request_ac_block`).
- Ordine delle modifiche: prima aggiornare logica .txt, poi YAML, infine plancia e README.

## 4️⃣ RIFERIMENTI
- **Logiche e priorità condivise** → `core/regole_core_logiche.md`.
- **Linee guida UI generali** → `core/regole_plancia.md`.
- **Dettagli di modulo** → rispettivo `<modulo>.txt` e `<modulo>_plancia_regole.txt`.
